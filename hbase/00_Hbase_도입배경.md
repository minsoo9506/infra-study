# HBase

## HBase 도입 배경

### 기존 시스템의 한계

#### 1. RDBMS의 확장성 문제
전통적인 관계형 데이터베이스(MySQL, Oracle 등)는 **수직 확장(Scale-Up)** 에 의존한다. 데이터가 수십억 건 이상으로 증가하면:
- 단일 서버의 CPU, 메모리, 디스크 I/O에 병목이 발생한다
- 샤딩(Sharding)으로 수평 확장이 가능하지만, 애플리케이션 레벨에서 라우팅 로직을 직접 관리해야 한다
- JOIN, 트랜잭션 등 RDBMS의 핵심 기능이 샤딩 환경에서 제약된다

#### 2. HDFS 단독 사용의 한계
Hadoop의 HDFS는 대용량 데이터를 안정적으로 저장할 수 있지만:
- **배치 처리에 최적화**되어 있어 실시간 랜덤 읽기/쓰기가 불가능하다
- 파일 단위 접근만 가능하며, 특정 레코드를 빠르게 조회하는 것이 어렵다
- 데이터 수정(update)이 불가능하고, append만 지원한다

### HBase가 해결하는 문제

#### 1. 대규모 데이터의 실시간 랜덤 액세스
HBase는 HDFS 위에서 동작하면서도 **밀리초 단위의 랜덤 읽기/쓰기**를 제공한다. Google의 Bigtable 논문(2006)을 기반으로 설계되었다.

#### 2. 자동 수평 확장 (Auto Sharding)
- 테이블을 **Region** 단위로 자동 분할한다
- Region이 일정 크기를 초과하면 자동으로 split되어 여러 RegionServer에 분산된다
- 애플리케이션이 샤딩 로직을 관리할 필요가 없다

#### 3. 유연한 스키마 (Schema-less)
- Column Family 기반의 **Wide Column Store** 구조를 채택한다
- 각 행(row)마다 서로 다른 컬럼을 가질 수 있다
- 스키마 변경 없이 새로운 컬럼을 언제든 추가할 수 있다

#### 4. 강한 일관성 (Strong Consistency)
NoSQL 중에서도 HBase는 **단일 행 수준의 강한 일관성**을 보장한다. 같은 row에 대한 읽기는 항상 최신 쓰기 결과를 반환한다. (Eventually Consistent인 Cassandra와 대비되는 특성)

### HBase의 주요 사용 사례

| 사용 사례 | 설명 |
|-----------|------|
| 메시징 플랫폼 | Facebook 메신저가 HBase를 메시지 저장소로 사용 |
| 시계열 데이터 | 대규모 IoT 센서 데이터, 모니터링 메트릭 저장 |
| 사용자 활동 로그 | 클릭스트림, 검색 이력 등 대량의 이벤트 데이터 |
| 추천 시스템 | 사용자-아이템 관계 데이터의 빠른 조회 |

### 요약: 언제 HBase를 선택하는가

```
RDBMS로 충분한 경우          HBase가 적합한 경우
─────────────────────       ─────────────────────
- 데이터 수억 건 이하          - 데이터 수십억 건 이상
- 복잡한 JOIN/트랜잭션 필요     - 단순 Key-Value 조회 패턴
- 강력한 SQL 기능 필요         - 실시간 랜덤 읽기/쓰기 필요
- 정형화된 스키마              - 유연한 스키마 필요
                             - HDFS 에코시스템 활용
```

HBase는 **"HDFS의 저장 안정성 + 실시간 랜덤 액세스"** 라는 조합이 필요할 때 도입을 고려하는 것이 적절하다.
