# Kafka 주요 특징

## 1. 분산 스트리밍 플랫폼
Kafka는 대용량 실시간 데이터 스트리밍을 처리하기 위한 분산 이벤트 스트리밍 플랫폼이다.

## 2. 높은 처리량(High Throughput)
- 배치 처리, 압축, 순차 I/O(Sequential I/O) 등을 활용하여 초당 수백만 건의 메시지를 처리할 수 있다.
- 프로듀서는 메시지를 배치로 묶어 전송하고, 브로커는 디스크에 순차적으로 기록하여 디스크 I/O를 최적화한다.

## 3. 내구성(Durability) & 내결함성(Fault Tolerance)
- 메시지를 디스크에 영속적으로 저장하며, 파티션 복제(Replication)를 통해 브로커 장애 시에도 데이터 유실을 방지한다.
- ISR(In-Sync Replica) 메커니즘으로 리더-팔로워 간 동기화를 관리한다.

## 4. 확장성(Scalability)
- 브로커를 수평 확장하고, 토픽을 파티션으로 분할하여 병렬 처리가 가능하다.
- 컨슈머 그룹을 통해 여러 컨슈머가 파티션을 나누어 소비한다.

## 5. Pub/Sub 메시징 모델
- 프로듀서가 토픽에 메시지를 발행하고, 컨슈머가 토픽을 구독하는 구조이다.
- 메시지는 소비 후에도 retention 기간 동안 유지되어 여러 컨슈머가 독립적으로 읽을 수 있다.

## 6. 순서 보장
- 파티션 단위로 메시지 순서가 보장된다. 같은 키를 가진 메시지는 동일 파티션으로 라우팅된다.

## 7. Pull 기반 소비
- 컨슈머가 자신의 속도에 맞춰 메시지를 가져가는(pull) 방식으로, 백프레셔(backpressure)를 자연스럽게 처리한다.

## 8. 풍부한 에코시스템
- **Kafka Connect**: 외부 시스템과의 데이터 연동 (DB, S3, Elasticsearch 등)
- **Kafka Streams**: 자바 기반 스트림 처리 라이브러리
- **Schema Registry**: Avro/Protobuf 스키마 관리

## Kafka 4.0 - 가장 큰 전환점

### 1. ZooKeeper 완전 제거, KRaft 전면 전환
- ZooKeeper 의존성이 완전히 제거되었다.
- 메타데이터를 `@metadata` 내부 토픽에 저장하며, 컨트롤러 노드 쿼럼이 이를 복제한다.
- 운영 복잡도가 크게 감소한다 (별도 ZooKeeper 클러스터 불필요).

### 2. Share Groups - 큐(Queue) 시맨틱 지원 (KIP-932, Early Access)
- 기존 Kafka는 파티션 단위로 컨슈머를 할당했지만, Share Group은 **같은 파티션을 여러 컨슈머가 협력적으로 소비**할 수 있다.
- 메시지를 **개별적으로 ACK/재시도** 가능하여, 전통적인 메시지 큐(RabbitMQ 등)와 유사한 워크 큐 패턴을 Kafka에서 직접 구현할 수 있다.
- 아직 Early Access 단계로 프로덕션 사용은 권장되지 않는다.

### 3. 새로운 컨슈머 리밸런스 프로토콜 (KIP-848)
- 리밸런스 로직을 **브로커 측으로 이동**하여, 기존의 "Stop-the-World" 문제(새 컨슈머 참여 시 전체 컨슈머가 일시 정지)를 해결했다.
- 대규모 클러스터에서 리밸런스 속도와 안정성이 크게 향상되었다.

### 4. Java 버전 요구사항 변경
- 클라이언트/Streams: Java 11 이상 필수
- 브로커/Connect/Tools: Java 17 이상 필수
- 로깅이 log4j2(YAML 설정)로 전환되었다.
