- kafka v4 기준 설명

# 카프카 리플리케이션

## 리플리케이션 개념과 역할

Kafka는 브로커 장애에도 데이터를 유실하지 않기 위해 파티션 단위로 **복제(replication)**를 수행한다. 토픽 생성 시 `replication.factor`를 지정하면 각 파티션이 해당 수만큼의 복제본(replica)을 가진다.

```
예시) 토픽 "orders", 파티션 3개, replication.factor=3

파티션 0: 복제본 3개 (브로커1, 브로커2, 브로커3)
파티션 1: 복제본 3개 (브로커2, 브로커3, 브로커1)
파티션 2: 복제본 3개 (브로커3, 브로커1, 브로커2)

→ 브로커 1대가 죽어도 나머지 2개의 복제본이 살아있으므로 데이터 유실 없음
→ replication.factor=3이면 최대 2대의 브로커 장애까지 견딜 수 있음
```

### 리플리케이션의 트레이드오프

| | replication.factor 높음 | replication.factor 낮음 |
|---|---|---|
| **내구성** | 브로커 다수 장애에도 데이터 보존 | 장애 시 데이터 유실 위험 |
| **디스크** | 복제본만큼 디스크 사용량 증가 | 디스크 절약 |
| **네트워크** | 팔로워 복제 트래픽 증가 | 네트워크 부하 적음 |
| **쓰기 지연** | `acks=all` 시 모든 복제본 확인 대기 | 응답이 빠름 |

> **실무 권장**: `replication.factor=3`이 대부분의 운영 환경에서 표준이다. 2이면 브로커 1대 장애에만 대응 가능하고, 4 이상은 디스크·네트워크 비용 대비 이점이 적다.

## 리더와 팔로워

각 파티션의 복제본은 **리더(Leader)** 1개와 **팔로워(Follower)** N-1개로 구성된다.

```
파티션 0의 복제본 구조 (replication.factor=3)

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   브로커 1     │     │   브로커 2     │     │   브로커 3     │
│              │     │              │     │              │
│  파티션0      │     │  파티션0      │     │  파티션0      │
│  [리더] ◀────────── [팔로워]        │     │  [팔로워]     │
│         ◀──────────────────────────────── │              │
│              │     │              │     │              │
│  offset: 0~7 │     │  offset: 0~7 │     │  offset: 0~7 │
└──────────────┘     └──────────────┘     └──────────────┘
       ▲  │
       │  │
  프로듀서  컨슈머
  (쓰기)  (읽기)

→ 모든 읽기·쓰기는 리더를 통해서만 이루어진다
→ 팔로워는 리더의 데이터를 지속적으로 복제(fetch)한다
→ 리더 장애 시 팔로워 중 하나가 새 리더로 승격된다
```

### 리더의 역할
- 프로듀서의 **쓰기 요청**을 처리하고 로컬 로그에 기록한다.
- 컨슈머의 **읽기 요청**을 처리한다.
- 팔로워의 Fetch 요청에 응답하여 데이터를 제공한다.
- ISR(In-Sync Replicas) 목록을 관리한다.

### 팔로워의 역할
- 리더에게 주기적으로 **Fetch 요청**을 보내 데이터를 복제한다.
- 팔로워는 컨슈머와 동일한 방식(pull)으로 리더의 로그를 읽어온다.
- 리더 장애 시 ISR에 포함된 팔로워가 **새 리더로 승격**된다.
- 팔로워는 클라이언트(프로듀서/컨슈머)의 요청을 직접 처리하지 않는다.

## 복제 유지와 커밋

### ISR (In-Sync Replicas)

ISR은 리더와 **동기화가 유지되고 있는 복제본의 집합**이다. 리더 자신도 ISR에 포함된다.

```
예시) replication.factor=3인 파티션

정상 상태:
  ISR = {브로커1(리더), 브로커2(팔로워), 브로커3(팔로워)}

브로커3이 느려져서 복제가 지연된 경우:
  ISR = {브로커1(리더), 브로커2(팔로워)}
  → 브로커3은 ISR에서 제거됨 (OSR로 이동)
  → 브로커3이 다시 따라잡으면 ISR에 복귀
```

### ISR 제거 조건

팔로워가 `replica.lag.time.max.ms`(기본 30초) 동안 리더에게 Fetch 요청을 보내지 않거나, Fetch 요청을 보냈지만 리더의 최신 오프셋을 따라잡지 못하면 ISR에서 제거된다.

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `replica.lag.time.max.ms` | 30000 (30초) | 팔로워가 이 시간 내에 리더를 따라잡지 못하면 ISR에서 제거 |

### 커밋과 하이워터마크 (High Watermark)

Kafka에서 **커밋**이란 ISR의 모든 복제본이 해당 메시지를 성공적으로 복제한 상태를 의미한다. 커밋된 메시지의 경계를 나타내는 것이 **하이워터마크(High Watermark, HW)**이다.

```
리더 로그:     [0] [1] [2] [3] [4] [5] [6] [7]
팔로워A 로그:  [0] [1] [2] [3] [4] [5] [6]
팔로워B 로그:  [0] [1] [2] [3] [4] [5]
                                       ↑           ↑
                                       HW          LEO(리더)
                                    (offset=5)   (offset=7)

HW (High Watermark) = 5
  → ISR 전체가 복제 완료한 마지막 오프셋
  → 컨슈머는 HW까지의 메시지만 읽을 수 있다
  → offset 6, 7은 아직 커밋되지 않았으므로 컨슈머에게 보이지 않음

LEO (Log End Offset) = 각 복제본의 마지막 오프셋 + 1
  → 리더 LEO=8, 팔로워A LEO=7, 팔로워B LEO=6
```

### 커밋의 동작 흐름

```
1. 프로듀서 → 리더: 메시지 전송 (offset 6, 7)
2. 리더: 로컬 로그에 기록, LEO=8
3. 팔로워A → 리더: Fetch 요청 (offset 6부터)
4. 리더 → 팔로워A: offset 6, 7 전달
5. 팔로워A: 로컬 로그에 기록, LEO=8
6. 팔로워B → 리더: Fetch 요청 (offset 6부터)
7. 리더 → 팔로워B: offset 6, 7 전달
8. 팔로워B: 로컬 로그에 기록, LEO=8
9. 리더: 모든 ISR의 LEO가 8 이상 → HW를 8로 갱신
10. 컨슈머: HW=8까지의 메시지를 읽을 수 있음
```

### acks 설정과 커밋의 관계

| acks | 프로듀서 응답 시점 | 커밋(HW) 보장 |
|------|-----------------|-------------|
| `0` | 전송 즉시 (응답 안 기다림) | 보장 안 됨 |
| `1` | 리더 로컬 기록 후 | 보장 안 됨 (리더 장애 시 유실 가능) |
| `all` | ISR 전체 복제 완료 후 | **보장됨** (HW까지 복제 확인) |

> `acks=all` + `min.insync.replicas=2`를 함께 사용하면, 최소 2개의 ISR이 메시지를 복제해야 프로듀서에게 성공 응답을 보낸다. ISR이 2개 미만이면 쓰기가 거부된다(`NotEnoughReplicasException`).

## 리더와 팔로워의 리플리케이션 동작

팔로워는 리더에게 **Fetch 요청**을 보내 데이터를 가져온다. 이 과정에서 리더는 팔로워의 복제 진행 상황을 추적하고 하이워터마크를 갱신한다.

### 상세 복제 흐름

```
시간 흐름 →

[t1] 프로듀서가 리더에 메시지 m1 전송
     리더:    [m0][m1]  LEO=2, HW=1
     팔로워A: [m0]      LEO=1
     팔로워B: [m0]      LEO=1

[t2] 팔로워A가 Fetch(offset=1) 요청
     → 리더가 m1을 응답하면서 현재 HW=1도 함께 전달
     팔로워A: [m0][m1]  LEO=2, HW=1 (리더가 알려준 HW로 갱신)

[t3] 팔로워B가 Fetch(offset=1) 요청
     → 리더는 팔로워B의 Fetch offset=1을 보고 "팔로워B는 offset 0까지 가지고 있다"고 파악
     → 리더가 m1을 응답하면서 현재 HW=1도 함께 전달
     → 리더: 모든 ISR(A, B)의 LEO ≥ 2 확인 → HW=2로 갱신
     팔로워B: [m0][m1]  LEO=2, HW=1

[t4] 다음 Fetch 시 팔로워들이 리더의 HW=2를 받아 자신의 HW도 갱신
     팔로워A: HW=2
     팔로워B: HW=2
```

### 핵심: 하이워터마크 전파의 지연

위 흐름에서 중요한 점은 **팔로워의 HW 갱신이 리더보다 한 라운드 늦다**는 것이다.

```
리더가 HW를 갱신하는 시점:
  → 팔로워의 Fetch 요청을 받을 때, 모든 ISR의 LEO를 확인하여 갱신

팔로워가 HW를 갱신하는 시점:
  → 리더로부터 Fetch 응답을 받을 때, 응답에 포함된 리더의 HW로 갱신
  → 따라서 리더가 HW를 올린 직후에는 팔로워의 HW가 아직 이전 값

→ 이 시간차 동안 리더가 장애 나면 데이터 불일치 가능성
→ 이 문제를 해결하기 위해 "리더 에포크(Leader Epoch)"가 도입됨
```

## 리더 에포크와 복구

### 리더 에포크 (Leader Epoch)

리더 에포크는 리더가 변경될 때마다 증가하는 **단조 증가 정수**이다. 각 에포크는 해당 리더가 쓰기를 시작한 오프셋과 함께 기록된다.

```
Leader Epoch 기록 예시:

에포크 0: 리더=브로커1, 시작 오프셋=0
  → 브로커1이 offset 0~5 기록

에포크 1: 리더=브로커2, 시작 오프셋=6  (브로커1 장애로 리더 교체)
  → 브로커2가 offset 6~10 기록

에포크 2: 리더=브로커1, 시작 오프셋=11  (브로커1 복구 후 다시 리더)
  → 브로커1이 offset 11~ 기록
```

### 리더 에포크가 없을 때의 문제 (하이워터마크만 사용)

```
문제 시나리오: HW 기반 복구의 데이터 유실

[초기 상태] replication.factor=2
리더(브로커1):  [m0][m1]  LEO=2, HW=2
팔로워(브로커2): [m0][m1]  LEO=2, HW=1  ← HW 전파 지연

[1단계] 팔로워(브로커2)가 재시작됨
  → 복구 시 자신의 HW(=1)까지만 신뢰하고, m1을 잘라냄(truncate)
  브로커2: [m0]  LEO=1

[2단계] 브로커2가 복구되기 전에 리더(브로커1)도 장애 발생
  → 브로커2가 새 리더로 승격됨 (유일한 ISR)
  새 리더(브로커2): [m0]  LEO=1, HW=1

[결과] m1이 영구 유실됨!
  → 프로듀서는 acks=all로 m1 전송 성공 응답을 받았지만 데이터가 사라짐
```

### 리더 에포크를 사용한 복구

```
리더 에포크 기반 복구:

[초기 상태] 에포크 0, 리더=브로커1
리더(브로커1):  [m0][m1]  LEO=2, HW=2, epoch=0
팔로워(브로커2): [m0][m1]  LEO=2, HW=1, epoch=0

[1단계] 팔로워(브로커2)가 재시작됨
  → HW로 자르지 않고, 리더에게 "에포크 0의 LEO가 몇이냐?" 질의
  → 리더 응답: "에포크 0의 LEO=2"
  → 브로커2의 LEO도 2이므로 자를 필요 없음!
  브로커2: [m0][m1]  LEO=2  ← 데이터 유지

[2단계] 이후 브로커1 장애 발생
  → 브로커2가 새 리더로 승격, epoch=1
  새 리더(브로커2): [m0][m1]  LEO=2, epoch=1

[결과] m1이 보존됨!
```

### 리더 에포크 복구 요약

| 상황 | HW 기반 (구버전) | 리더 에포크 기반 |
|------|----------------|----------------|
| 팔로워 재시작 | HW까지 truncate → 데이터 유실 가능 | 에포크 기반 LEO 확인 → 불필요한 truncate 방지 |
| 리더 변경 | HW 불일치로 일관성 깨질 수 있음 | 에포크 경계 기준으로 정확한 truncate 지점 결정 |
| 동시 장애 | 커밋된 메시지도 유실 가능 | 에포크 정보로 정확한 복구 가능 |

> 리더 에포크는 Kafka 0.11부터 도입되었으며, 현재(Kafka 4.0)에서는 기본으로 활성화되어 있다. `inter.broker.protocol.version` 설정이 0.11 이상이면 자동 적용된다.

# 컨트롤러

## 컨트롤러의 역할

컨트롤러는 Kafka 클러스터의 **메타데이터를 관리하고 파티션 리더 선출을 담당하는 브로커**이다. Kafka 4.0에서는 ZooKeeper가 완전히 제거되고 **KRaft(Kafka Raft)** 모드로만 동작한다.

```
Kafka 4.0 KRaft 클러스터 구성 예시

┌─────────────────────────────────────────────────────────────┐
│  KRaft 컨트롤러 쿼럼 (Raft 합의)                               │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                  │
│  │컨트롤러 1 │  │컨트롤러 2 │  │컨트롤러 3 │                  │
│  │(Active)  │  │(Standby) │  │(Standby) │                  │
│  └────┬─────┘  └──────────┘  └──────────┘                  │
│       │  메타데이터 변경 사항 전파                               │
└───────┼─────────────────────────────────────────────────────┘
        │
        ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 브로커 1   │  │ 브로커 2   │  │ 브로커 3   │
│ 파티션 리더 │  │ 파티션 리더 │  │ 파티션 리더 │
│ 파티션 팔로워│  │ 파티션 팔로워│  │ 파티션 팔로워│
└──────────┘  └──────────┘  └──────────┘
```

### KRaft 모드의 특징

| 항목 | ZooKeeper 모드 (구버전) | KRaft 모드 (Kafka 4.0) |
|------|----------------------|----------------------|
| 메타데이터 저장 | ZooKeeper에 외부 저장 | Kafka 내부 토픽(`__cluster_metadata`)에 저장 |
| 컨트롤러 선출 | ZooKeeper의 임시 노드로 선출 | Raft 합의 프로토콜로 선출 |
| 장애 복구 | ZooKeeper 의존 → 복구 느림 | 자체 Raft 로그로 빠른 복구 |
| 운영 복잡도 | Kafka + ZooKeeper 이중 관리 | Kafka만 관리 |
| 메타데이터 일관성 | 비동기 전파 → 일시적 불일치 가능 | Raft 로그 기반 → 강한 일관성 |
| 확장성 | 파티션 수십만 개에서 한계 | 수백만 파티션까지 확장 가능 |

### 컨트롤러의 주요 기능

**1. 파티션 리더 선출**
```
브로커 장애 발생 시 리더 선출 흐름:

1. Active 컨트롤러가 브로커 장애를 감지
2. 해당 브로커가 리더였던 모든 파티션에 대해 새 리더 선출
3. ISR 목록에서 첫 번째 복제본을 새 리더로 선출 (기본 정책)
4. 새 리더 정보를 메타데이터 로그에 기록
5. 모든 브로커에 LeaderAndIsr 요청으로 변경 사항 전파
6. 프로듀서/컨슈머가 메타데이터 갱신 후 새 리더로 요청
```

**2. 클러스터 메타데이터 관리**
- 토픽, 파티션, 복제본 할당 정보
- 브로커 등록/해제 상태
- ACL(접근 제어 목록)
- 설정 변경 이력

**3. 브로커 헬스 체크**
- 각 브로커는 컨트롤러에 주기적으로 하트비트를 전송한다.
- `broker.session.timeout.ms`(기본 9초) 내에 하트비트가 없으면 해당 브로커를 장애로 판단하고 파티션 리더를 재선출한다.

### Unclean Leader Election

ISR이 모두 다운된 경우, OSR(Out-of-Sync Replica)에서 리더를 선출할지 결정하는 설정이다.

```
ISR = {} (모두 다운), OSR = {브로커3}

unclean.leader.election.enable=true:
  → 브로커3을 리더로 선출
  → 서비스 가용성 확보, 하지만 데이터 유실 가능 (동기화 안 된 데이터)

unclean.leader.election.enable=false (기본값):
  → 리더를 선출하지 않음 → 해당 파티션 사용 불가
  → 데이터 무결성 보장, 하지만 서비스 중단
```

> **실무 권장**: 금융, 결제 등 데이터 유실이 치명적인 시스템에서는 `false`(기본값)를 유지한다. 로그, 메트릭 등 일부 유실이 허용되는 경우에만 `true`를 고려한다.

# 로그

Kafka의 메시지는 **로그(Log)**라는 구조로 디스크에 저장된다. 각 파티션은 하나의 로그이며, 로그는 여러 **세그먼트(Segment)** 파일로 나뉘어 관리된다.

```
파티션 디렉토리 구조 예시: /kafka-logs/orders-0/

orders-0/
├── 00000000000000000000.log        ← 세그먼트 파일 (메시지 데이터)
├── 00000000000000000000.index      ← 오프셋 인덱스
├── 00000000000000000000.timeindex  ← 타임스탬프 인덱스
├── 00000000000005242880.log        ← 두 번째 세그먼트 (시작 오프셋이 파일명)
├── 00000000000005242880.index
├── 00000000000005242880.timeindex
├── 00000000000010485760.log        ← 세 번째 세그먼트 (Active Segment)
├── 00000000000010485760.index
├── 00000000000010485760.timeindex
└── leader-epoch-checkpoint          ← 리더 에포크 정보
```

### 세그먼트의 구조

```
세그먼트 = .log + .index + .timeindex

.log 파일:
  실제 메시지(레코드 배치)가 순차적으로 저장된다.
  → 순차 쓰기(sequential write)만 발생하므로 디스크 I/O 효율이 높다.

.index 파일 (오프셋 인덱스):
  오프셋 → .log 파일 내 물리적 위치(바이트 오프셋) 매핑
  → 특정 오프셋의 메시지를 빠르게 찾기 위한 인덱스
  → 모든 오프셋이 아닌 일정 간격(log.index.interval.bytes)마다 기록 (희소 인덱스)

.timeindex 파일 (타임스탬프 인덱스):
  타임스탬프 → 오프셋 매핑
  → 특정 시간 기준으로 메시지를 찾을 때 사용 (offsetsForTimes API)
```

### Active Segment

- 가장 마지막 세그먼트는 **Active Segment**로, 현재 쓰기가 진행 중인 세그먼트이다.
- Active Segment는 삭제/컴팩션 대상에서 **제외**된다.
- 새 세그먼트가 생성되는 조건:
  - `log.segment.bytes`(기본 1GB)를 초과하거나
  - `log.roll.ms` / `log.roll.hours`(기본 7일) 시간이 경과

## 로그 세그먼트 삭제

오래된 세그먼트를 **삭제**하여 디스크 공간을 확보하는 정책이다. `log.cleanup.policy=delete`(기본값)일 때 동작한다.

### 삭제 조건

```
세그먼트 삭제가 발생하는 조건 (OR 조건):

1. 시간 기반 (retention.ms)
   → 세그먼트의 마지막 메시지 타임스탬프 기준으로 retention 기간 초과
   → 기본값: 7일 (604800000ms)

2. 크기 기반 (retention.bytes)
   → 파티션 전체 로그 크기가 설정값을 초과하면 오래된 세그먼트부터 삭제
   → 기본값: -1 (무제한)

주의: Active Segment는 조건을 충족해도 삭제되지 않음
```

### 주요 설정

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `log.cleanup.policy` | `delete` | 로그 정리 방식. `delete`, `compact`, `delete,compact` |
| `log.retention.ms` | 604800000 (7일) | 메시지 보존 기간 |
| `log.retention.bytes` | -1 (무제한) | 파티션당 최대 로그 크기 |
| `log.segment.bytes` | 1073741824 (1GB) | 세그먼트 파일 최대 크기 |
| `log.retention.check.interval.ms` | 300000 (5분) | 삭제 대상 세그먼트를 확인하는 주기 |

### 삭제 동작 흐름

```
예시) retention.ms=3일, 현재 시각 1월 10일

세그먼트 1: 마지막 메시지 1월 5일 → 5일 경과 → 삭제 대상 ✓
세그먼트 2: 마지막 메시지 1월 7일 → 3일 경과 → 삭제 대상 ✓ (정확히 3일)
세그먼트 3: 마지막 메시지 1월 9일 → 1일 경과 → 유지
세그먼트 4: (Active Segment, 현재 쓰기 중) → 유지 (삭제 불가)

삭제 과정:
1. 대상 세그먼트의 .log 파일명에 .deleted 접미사 추가
2. .index, .timeindex 파일도 동일하게 .deleted 처리
3. file.delete.delay.ms(기본 60초) 후 실제 파일 삭제
```

## 로그 세그먼트 컴팩션

컴팩션은 **같은 키를 가진 메시지 중 최신 값만 남기고** 이전 값을 제거하는 정책이다. `log.cleanup.policy=compact`일 때 동작한다.

### 컴팩션의 개념

```
컴팩션 전:
  offset 0: key=A, value="v1"
  offset 1: key=B, value="v1"
  offset 2: key=A, value="v2"   ← key=A 업데이트
  offset 3: key=C, value="v1"
  offset 4: key=B, value="v2"   ← key=B 업데이트
  offset 5: key=A, value="v3"   ← key=A 다시 업데이트

컴팩션 후:
  offset 3: key=C, value="v1"   ← 유일한 값이므로 유지
  offset 4: key=B, value="v2"   ← key=B의 최신 값
  offset 5: key=A, value="v3"   ← key=A의 최신 값

→ 각 키의 최신 상태(snapshot)만 남긴다
→ 오프셋 번호는 변경되지 않는다 (중간 오프셋이 비게 됨)
```

### 컴팩션의 용도

| 용도 | 설명 |
|------|------|
| **__consumer_offsets** | 컨슈머 그룹별 마지막 커밋 오프셋만 유지 |
| **설정 저장소** | 각 설정 키의 최신 값만 유지 (e.g., Kafka Connect 설정) |
| **CDC (Change Data Capture)** | DB 테이블의 각 행(PK=key)의 최신 상태만 유지 |
| **이벤트 소싱** | 엔티티별 최신 상태 스냅샷 |

### 컴팩션 동작 원리

```
로그 영역 구분:

                    Clean 영역              Dirty 영역
               (이미 컴팩션 완료)        (컴팩션 아직 안 됨)
              ┌──────────────────┐  ┌──────────────────────┐
              │ 키 중복 없음       │  │ 키 중복 있을 수 있음    │
              │ (최신 값만 존재)    │  │                      │
              └──────────────────┘  └──────────────────────┘
                                    ↑
                              Cleaner Point
                         (컴팩션 시작 위치)

컴팩션 수행:
1. Log Cleaner 스레드가 Dirty 영역을 스캔
2. 각 키의 최신 오프셋을 맵에 기록
3. Dirty 영역을 순회하면서 최신이 아닌 레코드를 제거
4. 컴팩션된 결과를 새 세그먼트로 작성
5. 기존 세그먼트를 새 세그먼트로 교체
```

### Tombstone (삭제 마커)

키에 대한 데이터를 완전히 삭제하려면 **value=null**인 메시지를 보낸다. 이를 **톰스톤(tombstone)**이라 한다.

```
offset 10: key=A, value=null   ← 톰스톤

→ 컴팩션 시 key=A의 이전 값들이 모두 제거됨
→ 톰스톤 자체도 delete.retention.ms(기본 24시간) 이후에 제거됨
→ 최종적으로 key=A에 대한 모든 흔적이 사라짐
```

### 컴팩션 관련 주요 설정

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `log.cleanup.policy` | `delete` | `compact`로 설정 시 컴팩션 활성화 |
| `log.cleaner.min.compaction.lag.ms` | 0 | 메시지 생성 후 최소 이 시간이 지나야 컴팩션 대상 |
| `log.cleaner.max.compaction.lag.ms` | ∞ | 이 시간이 지나면 반드시 컴팩션 수행 |
| `min.cleanable.dirty.ratio` | 0.5 | Dirty 비율이 이 값을 넘으면 컴팩션 시작 |
| `delete.retention.ms` | 86400000 (24시간) | 톰스톤 보존 기간 |

### delete + compact 혼합 정책

```
log.cleanup.policy=delete,compact

→ 컴팩션으로 각 키의 최신 값만 유지하면서
→ retention.ms가 지난 메시지는 삭제
→ "최신 상태는 유지하되, 너무 오래된 것은 삭제"하는 전략
```

> **실무 요약**: 일반적인 이벤트/로그 토픽은 `delete` 정책으로 보존 기간만 설정하면 충분하다. 상태 저장소(consumer offsets, CDC, 설정 동기화)로 사용하는 토픽에는 `compact` 정책을 적용하여 키별 최신 상태만 유지한다.
